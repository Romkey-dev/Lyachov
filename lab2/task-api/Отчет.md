## 4. Реализованные компоненты

### 4.1 Утилиты работы с файлами (`src/utils/fileOperations.js`)

**Функции:**
- `initializeDataFile()` - инициализация JSON-файла при запуске
- `readData()` - чтение данных из файла
- `writeData(data)` - запись данных в файл
- `getNextId()` - генерация следующего ID

**Особенности:**
- Асинхронная обработка операций с файлами
- Автоматическое создание файла с начальными данными
- Обработка ошибок при работе с файловой системой

### 4.2 Middleware валидации (`src/middleware/validation.js`)

**Реализованные схемы Joi:**

**createTaskSchema:**
- `title` - 3-100 символов, обязательно
- `description` - до 500 символов
- `category` - перечисление (work, personal, shopping, health)
- `priority` - число 1-5
- `dueDate` - дата в будущем

**updateTaskSchema:**
- Те же поля, что и createTaskSchema, но все опциональны

**Middleware функции:**
- `validateCreateTask` - валидация при создании
- `validateUpdateTask` - валидация при обновлении
- `validateId` - валидация ID из URL

### 4.3 Обработчик ошибок (`src/middleware/errorHandler.js`)

**Функции:**
- `notFoundHandler` - обработка 404 ошибок
- `errorHandler` - централизованная обработка всех ошибок

**Особенности:**
- Структурированные JSON ответы об ошибках
- Логирование с timestamp
- Разные уровни детальности в зависимости от окружения

### 4.4 API эндпоинты (`src/routes/tasks.js`)

#### **CRUD операции:**

1. **GET /api/tasks** - получить все задачи
   - Параметры: category, completed, priority, sortBy, page, limit
   - Фильтрация по категории, статусу, приоритету
   - Сортировка в прямом/обратном порядке
   - Пагинация (по умолчанию page=1, limit=10)

2. **POST /api/tasks** - создать новую задачу
   - Валидация входных данных
   - Автогенерация ID и UUID
   - Установка timestamps (createdAt, updatedAt)

3. **GET /api/tasks/:id** - получить задачу по ID
   - Валидация ID
   - Возврат 404 если задача не найдена

4. **PUT /api/tasks/:id** - обновить задачу
   - Частичное обновление (только переданные поля)
   - Обновление updatedAt
   - Валидация данных

5. **PATCH /api/tasks/:id/complete** - отметить как выполненную
   - Специализированный endpoint для изменения статуса

6. **DELETE /api/tasks/:id** - удалить задачу
   - Проверка существования
   - Удаление из массива и сохранение

#### **Дополнительные функции:**

7. **GET /api/tasks/stats/summary** - статистика
   - Общее количество задач
   - Выполненные и невыполненные
   - Просроченные задачи
   - Распределение по категориям и приоритетам

8. **GET /api/tasks/search/text?q=...** - поиск
   - Поиск по title и description
   - Регистронезависимый поиск
   - Минимум 2 символа в запросе

## 6. Сравнение подходов: Express vs FastAPI (из части 1)

| Критерий | Express (Node.js) | FastAPI (Python) |
|----------|------------------|------------------|
| **Язык** | JavaScript | Python |
| **Синтаксис** | Более многословный | Лаконичный, декораторы |
| **Валидация** | Joi (внешняя库) | Pydantic (встроена) |
| **Type hints** | Слабая типизация | Строгая типизация |
| **Документация** | Swagger (нужна настройка) | Автоматическая (встроена) |
| **Производительность** | Высокая | Очень высокая (async) |
| **Асинхронность** | Встроена (async/await) | Встроена (async/await) |
| **Экосистема** | Огромная (npm) | Мощная (pip) |
| **Кривая обучения** | Средняя | Средняя |
| **Для production** | Отличный выбор | Отличный выбор |

### 6.1 Пример эквивалентного кода

**Express:**
```javascript
router.post('/', validateCreateTask, async (req, res, next) => {
  try {
    const data = await readData();
    const newTask = { ...req.body, id: await getNextId() };
    data.tasks.push(newTask);
    await writeData(data);
    res.status(201).json({ success: true, data: newTask });
  } catch (error) {
    next(error);
  }
});
```

**FastAPI:**
```python
@app.post("/tasks")
async def create_task(task: TaskCreate):
    new_task = Task(**task.dict(), id=next_id)
    tasks_db.append(new_task)
    return new_task
```

**Вывод:** FastAPI более лаконичен благодаря встроенной валидации и автоматической документации, но Express больше контроля дает разработчику.

## 7. Преимущества файлового хранения данных

### Для этого задания файловое хранение подходит, потому что:

1. **Простота реализации**
   - Не нужно настраивать БД
   - Нет подключений и пула соединений
   - Готово работает на любой машине

2. **Достаточно для малых объемов**
   - Для учебного проекта идеально
   - Можно хранить сотни/тысячи задач

3. **Быстрота разработки**
   - Сосредоточение на логике API
   - Минимум инфраструктуры

4. **Портативность**
   - Данные в одном JSON-файле
   - Легко делиться проектом

### Ограничения и когда переходить на БД:

| Проблема | Решение |
|----------|---------|
| Конкурентные записи | Использовать PostgreSQL, MongoDB |
| Большой объем данных (>10K задач) | Реальная БД с индексами |
| Множество пользователей | БД + аутентификация |
| Сложные запросы (JOIN, агрегация) | SQL, NoSQL |
| Высокая нагрузка | Кэширование (Redis) + БД |

**Express vs FastAPI:**
- Express более гибкий, требует больше настройки
- FastAPI более "батарейки в комплекте", но меньше контроля
- Оба отличны для production при правильной настройке

**Express vs Django:**
- Express быстрее для микросервисов
- Django мощнее для монолитов с множеством встроенных функций
